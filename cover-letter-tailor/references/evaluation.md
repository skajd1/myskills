# 자기소개서 평가 기준

## 기업/JD 분석 기준

- 기업의 사업 영역, 고객, 제품/서비스, 기술 스택, 인프라 운영 특성을 파악한다.
- JD에서 반복되는 동사와 명사를 평가 키워드로 본다. 예: 운영, 구축, 자동화, 장애 대응, 클라우드, GPU, MLOps, 보안, 데이터 정합성.
- 직무가 요구하는 성과를 한 문장으로 정의한다. 예: "대규모 AI 인프라를 안정적으로 운영하고 배포 효율을 높이는 역할".
- 답변의 마지막 문단은 이 성과 정의와 연결한다.

## 기존 경험 선택 기준

| 문항 유형 | 우선 소재 |
| --- | --- |
| 지원 동기 | 회사 사업/JD와 연결되는 기술 관심, 입사 후 기여 계획 |
| 직무 역량 | Linux, Shell script, CI/CD, Docker/K8s, GPU/TensorRT, MariaDB 이중화, 문서화 경험 |
| 협업 | Jira/Confluence 전환, 가이드 제작, 문서 구조 표준화 |
| 성취/성장 | 학부 연구생 딥러닝 과제, 최하위에서 최고 정확도 달성 |
| 프로젝트 | OO증권 비대면 계좌 개설, 얼굴인식 솔루션, DR/DB 이중화 |
| 가치관 | 개인의 지식을 공동의 지식으로 만드는 태도 |

## 문장 품질 기준

- 한 문장에 메시지를 하나만 담는다.
- "기여하고 싶습니다"보다 "기여하겠습니다"처럼 실행 의지를 분명히 쓴다.
- 같은 문단에서 "이를 통해", "또한", "특히"를 반복하지 않는다.
- "같이 같이", "각 서비스별" 같은 중복 표현을 고친다.
- 기술명은 일관되게 쓴다. 예: CI/CD, Shell script, MariaDB Master-Master, TensorRT, Docker, Kubernetes.
- 과장된 표현은 근거가 있을 때만 쓴다. 예: "국내 최대", "유일", "글로벌 최고"는 출처 없이 단정하지 않는다.

## 기업 맞춤화 기준

- 기업명을 바꾸는 것만으로 맞춤화했다고 보지 않는다.
- 기업의 사업 맥락과 JD 요구 역량을 최소 한 번 이상 답변 본문에 자연스럽게 녹인다.
- 기존 경험은 유지하되, 마지막 20-30%는 대상 기업의 환경과 직무 성과로 변환한다.
- 다른 기업명이나 다른 산업의 표현이 남으면 제출 불가로 본다.

## 최종 산출물 기준

- Markdown 파일 하나에 기업/JD 분석과 문항별 최종 답변을 함께 둔다.
- 문항 원문과 제한 글자 수를 보존한다.
- 답변 제목은 대괄호 한 줄로 작성한다.
- 최종 답변에는 교정 전 초안이나 메모를 남기지 않는다.
