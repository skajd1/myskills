# 개인 경험 및 직무 역량 분석

이 문서는 현재 프로젝트에 정리된 자기소개서들을 바탕으로, 향후 기업 맞춤형 자기소개서 작성 시 재사용할 수 있는 경험, 직무 역량, 가치관, 문장 소재를 정리한 기준 문서입니다.

## 핵심 포지셔닝

- 지원 직무 축: 시스템 운영 엔지니어, 클라우드/AI 인프라 엔지니어, AI Platform 운영, 소프트웨어/인프라 운영
- 강점 키워드: 운영 자동화, 표준화, Linux 서버 관리, 폐쇄망 배포, AI 서비스 운영, GPU 런타임 최적화, CI/CD, DB 이중화, 문서화, 협업 체계화
- 차별화 포인트: 단순 개발보다 실제 운영 환경의 안정성, 가용성, 배포 일관성, 장애 대응 가능성을 중시하는 엔지니어
- 작성 톤: 기술 경험을 근거로 차분하게 설득하되, 마지막 문단에서 기업의 인프라/서비스 안정성과 연결하는 방식이 잘 맞음

## 직무 역량 분석

### 1. 시스템 운영 및 Linux 기반 인프라 관리

- Red Hat 계열 Linux 환경에서 서버 관리 경험이 있음.
- 사내 솔루션 서버 운영, 폐쇄망 설치, 환경 변수/ini 설정 등 실제 운영 환경의 배포 문제를 다룬 경험이 있음.
- 단순 기능 구현보다 운영 중 발생할 수 있는 설정 오류, 장애, 반복 작업을 줄이는 방향의 사고가 강함.

활용하기 좋은 직무:

- 시스템 운영 엔지니어
- 인프라 엔지니어
- 클라우드 운영
- SRE/DevOps 성격의 운영 직무

강조 문장 방향:

- "반복 작업을 표준화하고 자동화하여 운영 안정성을 높이는 엔지니어"
- "설정 하나가 서비스 전체에 영향을 주는 환경에서 휴먼 에러를 줄이는 운영 체계 구축"

### 2. 배포 자동화 및 표준화

- 수동으로 수정하던 ini 파일과 환경 변수 설정을 Shell script 기반 대화형 CLI로 자동화한 경험이 있음.
- 현장 엔지니어가 입력한 값이 자동으로 시스템에 반영되도록 하여 배포 오류 가능성을 낮춤.
- 배포 작업의 일관성을 확보하고 전체 배포 소요 시간을 50% 단축한 성과가 있음.

활용하기 좋은 문항:

- 직무 역량
- 문제 해결 경험
- 업무 효율화 경험
- 본인만의 강점

주의할 점:

- "자동화했습니다"에서 끝내지 말고, 왜 자동화가 필요했는지, 어떤 위험을 줄였는지, 결과가 무엇인지까지 써야 함.

### 3. AI 서비스 운영 및 GPU/MLOps 확장 가능성

- C++ 기반 AI 얼굴 인식 솔루션 운영 경험이 있음.
- TensorRT를 활용한 GPU 런타임 최적화 경험과 부하 테스트를 통한 GPU 자원 활용 경험이 있음.
- AI 모델 배포 환경, 컨테이너 패키징, CI/CD 경험을 MLOps로 확장해 설명하기 좋음.

활용하기 좋은 직무:

- AI Platform 운영
- MLOps
- AI 인프라
- 클라우드 AI 인프라 구축

강조 문장 방향:

- "모델 자체 개발보다 운영 환경에서 AI 서비스가 안정적으로 동작하도록 만드는 역량"
- "GPU 자원 상태를 모니터링하고 병목에 대응하는 AI Platform 운영 역량"

### 4. 데이터 가용성 및 장애 대응

- OO증권 비대면 계좌 개설 프로젝트에서 얼굴인식 솔루션 도입을 위해 개발/운영/DR 서버 총 4대의 인프라를 구축하고 담당함.
- 고객사 시연 중 서버 장애 시에도 서비스가 중단되지 않아야 한다는 요구사항을 받고, MariaDB Master-Master 복제와 GTID 기반 동기화 유지 방안을 적용함.
- 장애 상황을 가정한 모의 테스트로 데이터 유실 없이 서비스가 지속됨을 검증함.

활용하기 좋은 문항:

- 프로젝트 완수 경험
- 장애 대응 경험
- 고객 요구 대응
- 데이터 무결성/가용성
- 금융/보안/운영 직무

강조 문장 방향:

- "기능 구현을 넘어 운영 환경에서의 가용성과 데이터 무결성을 검증한 경험"
- "고객 요구를 기술 구조로 전환하고 장애 시나리오까지 검증한 경험"

### 5. RAG 기반 사내 지식 검색 시스템 개발

- 포트폴리오 대표 사례로 `Confluence / Jira RAG 시스템`을 구축함.
- 반복적으로 발생하던 "어디에 적혀 있지?"라는 업무 문제를 정의하고, Confluence 문서와 Jira 현황을 함께 검색해 근거 기반으로 답변하는 사내 지식 검색 시스템으로 전환함.
- 기술 스택은 FastAPI, Next.js, Ollama, Qdrant, LangChain, Confluence/Jira API이며, 포트폴리오 전체 기술 스택에는 Java, Python, Codex, MySQL, Docker, Kubernetes, Linux, GitHub Actions, GitLab CI/CD도 포함됨.
- 지식 수집 파이프라인은 원천 데이터 수집, 정제, 분할, 저장, 자동 갱신 흐름으로 설계하고 Qdrant 벡터 DB에 저장함.
- 질문 응답 파이프라인은 질문 유형 분기, Hybrid Search(BM25 + Vector Search), 출처 포함 LLM 답변 생성 구조로 설계함.
- 질문 유형에 따라 단일 정보 조회, 기간 요약, Jira 현황, 문서 기반 설명, 프로젝트 종합 등 검색 경로를 다르게 설계함.
- 사용자가 스페이스나 프로젝트를 직접 지정하는 `@멘션` 기능을 제공해 불필요한 문서가 섞이지 않도록 검색 범위를 좁힘.
- Confluence 문서와 Jira 상태가 계속 바뀌는 문제를 해결하기 위해 정해진 시간마다 자동 수집(ingest)을 실행하고, 새 수집이 성공했을 때만 기존 데이터를 교체하는 방식으로 최신성과 안정성을 함께 고려함.
- SSE 실시간 스트리밍을 적용해 전체 답변 완료 후 한 번에 표시하던 UX를 개선함. 기존에는 첫 화면 변화까지 30-40초가 걸릴 수 있었으나, 첫 문장을 5-8초 안에 보여주도록 개선함.
- 답변 근거 문서 표시 기능으로 문서 위치, Jira 상태, 담당자, 날짜를 함께 제공하여 답변 신뢰성을 높임.

활용하기 좋은 문항:

- 개발 프로젝트 경험
- AI/AX/DX 프로젝트 경험
- 문제 정의 및 해결 과정
- 고객/사용자 관점 서비스 개선
- RAG, LLM, 검색 시스템, 사내 지식 관리

강조 문장 방향:

- "AI 기능을 붙이는 데서 멈추지 않고, 반복 업무의 병목을 정의해 실제 업무 흐름을 개선한 경험"
- "검색 정확도, 최신성, 응답 속도, 근거 확인 가능성을 함께 고려해 운영 가능한 RAG 시스템을 설계한 경험"

### 6. AI 협업 및 개발 프로세스 설계

- 포트폴리오 보조 사례로 `How do I work with AI` 작업 방식을 정리함.
- AI에게 바로 일을 맡기기보다, 먼저 볼 문맥과 확인 기준을 정한 뒤 함께 작업하는 방식을 사용함.
- `docs/index.md`를 기준으로 AI가 참고할 문서 순서를 정하고, 작업 규칙, 시스템 이해, 코드 영역, 실행과 배포 문서를 분리함.
- 요구사항과 완료 기준을 먼저 정의한 뒤 구현하는 명세 기반 개발(SDD)을 지향함.
- 상위 모델이 전체 맥락을 관리하고, 구현/테스트/리뷰 에이전트가 단위 작업을 나누어 수행하는 구조를 활용함.
- 구현은 단위 기능 중심으로 나누고, TDD 기반 검증, 코드 리뷰, HTML 결과 확인을 통해 품질을 점검함.

활용하기 좋은 문항:

- 개발 방식
- 협업 방식
- AI 활용 경험
- 문제를 구조화하는 역량
- 품질 관리 및 검증 태도

강조 문장 방향:

- "AI를 단순 자동 생성 도구가 아니라, 명세와 검증 기준 안에서 생산성을 높이는 협업 도구로 활용"
- "요구사항, 제약 조건, 완료 기준을 먼저 정의해 작업 범위가 흐려지지 않도록 관리"

### 7. 문서화와 협업 체계화

- SharePoint에서 Jira/Confluence로 협업 툴을 전환할 때, 초기 세팅 및 이용 가이드를 제작함.
- 문서 디렉토리 구조를 표준화하고 Jira 티켓 생성 규칙을 수립함.
- 인프라 정보, 서버 용도, 네트워크 구성도, 리소스 관계 등을 Confluence에 문서화한 경험이 있음.

활용하기 좋은 문항:

- 협업 경험
- 가치관
- 조직 기여
- 갈등/혼선 해결
- 일하는 방식

강조 문장 방향:

- "개인의 지식을 공동의 지식으로 전환하는 사람"
- "특정 인원의 기억에 의존하지 않는 운영 체계를 만드는 사람"

### 8. 몰입과 성장 경험

- 학부 연구생 시절 딥러닝 기반 이미지 판별 과제를 수행함.
- 초기에는 Python 숙련도 부족으로 최하위 점수를 받았으나, 매일 알고리즘 문제 풀이와 PyTorch 모델 분석, 하이퍼파라미터/옵티마이저 실험 로그 기록을 통해 개선함.
- 최종적으로 연구실 역대 최고 정확도를 달성함.

활용하기 좋은 문항:

- 성장 과정
- 실패 극복
- 성취 경험
- 가치관
- 학업/연구 경험

강조 문장 방향:

- "부족함을 인정하고 원인을 분석해 성과로 전환하는 사람"
- "감이 아니라 로그와 실험을 통해 개선하는 데이터 기반 문제 해결자"

## 가치관 및 성향

### 조직의 동반 성장

- 개인의 경험과 지식을 팀 전체가 활용할 수 있는 자산으로 만드는 것을 중요하게 여김.
- 문서화, 가이드 제작, 표준화, 운영 매뉴얼화 경험과 잘 연결됨.

대표 문장:

> 개인의 지식을 공동의 지식으로 만들 때 조직의 시행착오를 줄이고 업무 효율을 높일 수 있다고 생각합니다.

### 목표 달성을 향한 몰입

- 실패나 초기 부족함을 회피하지 않고, 원인을 분석하고 반복 실험을 통해 성과를 만듦.
- 딥러닝 과제 경험과 가장 잘 연결됨.

대표 문장:

> 목표한 결과가 나올 때까지 원인을 분석하고 개선하는 태도는 제가 가진 가장 큰 강점입니다.

### 운영 안정성 중심의 사고

- 기능 구현 자체보다 실제 운영 환경에서 장애 없이 지속되는 구조를 중요하게 봄.
- DB 이중화, DR, 배포 자동화, 문서화 경험과 연결됨.

대표 문장:

> 운영 환경에서는 작은 설정 오류 하나가 서비스 전체에 영향을 줄 수 있기에, 반복 작업을 표준화하고 검증 가능한 구조로 만드는 것이 중요하다고 생각합니다.

## 보완이 필요한 지점

- AIOps, MLOps, Kubernetes, IaC는 향후 기여 계획으로는 좋지만 실제 직접 경험과 계획을 구분해야 함.
- 포트폴리오의 Docker, Kubernetes, GitHub Actions, GitLab CI/CD, MySQL, Java 등은 기술 스택으로 확인되지만, 자기소개서에서는 직접 기여 범위가 확실한 기능 중심으로 작성해야 함.
- "국내 최대", "국내 유일" 같은 기업 설명은 출처가 없으면 단정적으로 쓰지 않는 편이 안전함.
- 롯데이노베이트 `software(멤버스IS팀)2512.md`에는 KINX 표현이 남아 있으므로 재사용 시 반드시 기업명과 사업 맥락을 교체해야 함.
- "CICD"는 최종 제출문에서는 "CI/CD"로 통일하는 것을 권장함.
- "같이 같이", "각 서비스별" 등 중복 표현은 최종 교정 단계에서 제거해야 함.
